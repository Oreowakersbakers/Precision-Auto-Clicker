# Precision Auto Clicker Roadmap

This document captures future product direction. Completed implementation
history belongs in `CHANGELOG.md`; stable behavior belongs in `SPEC.md`; module
and platform details belong in `ARCHITECTURE.md`.

## Design North Star

Primary visual reference:

- `references/auto clicker visual reference.png`

Chosen direction: Option 1, the numbered Precision Console concept.

The app should feel like a native, calm Windows utility: clear, compact, and
confident. It should keep OP Auto Clicker's familiar mental model while improving
scan order, hierarchy, and operator feedback.

## Current Baseline

- Python/Tkinter desktop app with Win32 `ctypes` integration.
- Compact default UI with Interval, hotkey/advanced controls, Start/Stop, and
  footer metrics always visible.
- Click, Repeat, and Position controls live behind the labeled `Advanced`
  disclosure.
- Runtime hotkey changes are supported for plain keys.
- Clicking runs on a worker thread with high-resolution waitable timer support
  when available.
- Packaging is available through `Build-Exe.ps1`, `Build-Exe.bat`, and
  `PrecisionAutoClicker.spec`.

## Next Product Directions

### 1. Mini Always-On-Top Mode

Goal: add a tiny floating control surface for users who only need interval,
state, and Start/Stop while another app is in front.

Expected changes:

- Reuse the existing start/stop and active-hotkey state.
- Keep a clear path back to the full settings window.
- Do not start clicking automatically when entering or leaving mini mode.
- Document any always-on-top behavior in `SPEC.md` and `TEST_PLAN.md`.

Acceptance checks:

- Mini mode never hides Stop.
- Full mode and mini mode show the same running/stopped state.
- Closing either surface stops clicking when the app exits.

### 2. Record And Playback

Status: deferred and not visible in the current UI.

Goal: introduce macro recording carefully later, without confusing it with basic
auto clicking.

Expected changes:

- Define the macro data model and safety rules before adding controls.
- Do not show Record and Playback controls until the behavior is implemented or
  explicitly represented as unavailable.
- Keep playback stop behavior obvious and separate from ordinary clicking.

Acceptance checks:

- The app must never start recording silently.
- Playback must have an obvious stop path.
- Macro behavior must not change the core auto-clicker loop without explicit
  planning.

### 3. Profiles And Presets

Goal: make common configurations easy to reuse.

Expected changes:

- Add simple presets such as Default, QA Testing, and Clicker Game.
- Store profiles locally in a readable format only after the user approves
  persistence.
- Document any persistence format in `ARCHITECTURE.md`.

Acceptance checks:

- Defaults remain safe.
- Loading or saving a profile must not start clicking.
- Missing or malformed profile data falls back safely.

### 4. Timing Confidence

Goal: make performance understandable without exposing too much machinery.

Expected changes:

- Keep estimated CPS visible in the footer metrics strip.
- Keep jitter, CPS, CPU, uptime, and click count readable at minimum size.
- Add timing detail only when it directly helps operation or troubleshooting.

Acceptance checks:

- Stats update while clicking.
- UI remains responsive at fast intervals.
- Performance labels fit without overlap at minimum size.

### 5. Packaging And Distribution

Goal: make the app easy for a non-coder to run and straightforward to release.

Expected changes:

- Keep the repo-owned PyInstaller path working.
- Decide later whether to stay with Python/Tkinter long term or move to a more
  native Windows framework.
- Add release notes and checksum steps before broader sharing. (App icon
  done: `assets/app_icon.ico`/`.png`, generated by `packaging/make_icon.py`,
  wired into the window and the PyInstaller spec.)

Acceptance checks:

- Packaged app still uses Windows-native APIs.
- The elevated-app input caveat remains documented.
- Release steps exist before public distribution.

## Guardrails

- Do not change the click engine while doing visual work.
- Do not change timing strategy while doing visual work.
- Do not implement persistence, custom background behavior, or macro playback
  without a separate plan.
- Do not remove safety caveats.
- Keep the app usable for a non-coder: visible labels, obvious defaults, and an
  always-visible Stop path while clicking.
